npm@5规范

NPM是GitHub上最活跃的问题导向社区)，所以我们可以不断的接触到你的痛点和关注点。这意味着我们能够通过此去做更好的优化，以及决定什么事情是我们必须先去优化的。然而，由于用户和项目依赖者的数量巨大)，所以我们必须了解每个人的需求，然后不断的去做优化。

由于npm@4)的发布，负责NPM的客户端的团队一直在努力开发下一个主要版本”。NPM 4是为了清除以前版本的某些问题，但NPM 5将是一个更加期待的版本，所以我们尝试通过一种新方式征求你的反馈和担忧。

这个过程被分解成几块内容。让我们来谈谈他们,以及你怎么参与进来。

这就是 5!

在node@8出来之前，我们打算使“npm@5”有如下特点:

  • 一个更快的客户端，可以通过基准测试和分析工具来度量。
  • 一个很大的提高对于npm-shrinkwrap.json，NPM的版本锁定文件。特别是，封装打包项目本来是一个在开发部署之前用来最终使用的工具，但我们打算把此用于你的JavaScript应用程序的整个生命周期；
  • 新工具服务于npm注册

规格，或者，我们怎么到这里的？

“npm是怎么发展到此的？当然，这不是偶然的。

npm是一个快速迭代设计的产品以解决迫切的需求。在客户端中，人们所依赖的相当数量的东西的贡献者要么是Isaac开发的，要么是社区成员试图解决特定问题开发的。

例如npm shrinkwrap这个明显的例子，是Isaac和他的同学开发的 npm shrinkwrap 作为解决特定开发的问题作出的一个补丁。

这种快速迭代是npm作为一个开放项目成功的一部分，但它也面对着挑战。无设计文件或规范，开发者如果想去做一些工作，必须知道它是如何设计工作的。这是一个合乎逻辑的假设，但随着时间的推移，人们在这上面的假设，建立了越来越复杂的工作流。如何我们想不破坏特定人工作流程去做哪怕一点简单的特定的改变都是很困难的。

我从中得出了一个结论：我们已经认识到客户端不需要某种规格就能生存的地步是不可能的。

写下我们所期待的客户端，是用来帮助维护人员和开发人员解决困扰点，并使其更容易融入到你的项目中，去建设和维护的JavaScript应用程序。

从中得到的另一个结论:如果更多的人加入，我们的规范会更加好。

CLI团队的每一位成员都知道包管理的复杂性，但我们不是无所不知的。我们只看到了你的用例的一小部分-在大多数情况下，我们通过让你直接与其他人交谈来了解我们需要构建什么。

为了支持这一点，我们提出了一个过程，鼓励您帮助我们找出如何构建正确的功能以满足您的需求。

RFC流程

流程很简单:

  1. 一个或者多个客户端开发人员依照 npm’s own in-house process.起草一个粗略的文档

  2. 该草案公布在# RFC上，为我们团队内部审查。也让其他npm团队也能看到，以确保我们在做正确的事，这样规范才有意义的。（这不仅仅是工程——在产品设计、支持、文档和营销的反馈才是很重要的一点。）

  3. 该规范是在GitHub作为doc/specs/的一个新的子目录。随着时间的推移，这将增长到包括我们如何打算CLI的所有的核心组件的行为规范。

  4. 这将要求将社区审查至少两个完整的工作周，我们将公布新的草案通过NPM的每周社区新闻稿)。

  5. 在规范充分审查之后，我们将把所有的规范合并，然后将编写必要的代码来实现指定的特性。

这个过程不应该看起来特别新或令人兴奋，我们从其他项目中借用的。我们还试图平衡我们从项目的合作者和更广泛的社区获得良好反馈的需求，同时保持事情的进展。准备Node.js 8的发布，我们需要为NPM @ 5在2017年3月前完成重大更改。鉴于我们的雄心，这是一个紧迫的期限。

下一步是什么

好消息：我们准备好发起RFC的第一版了！

除了这个事情以外，Rebecca和Kat一直致力于研究一套如何变化，怎么让符号链接和本地依赖与压缩一起工作。今天至少他们其中的一个不再做这个工作了，所以他们的建议是很有趣的，至少有一点激进。我们会在每个人从寒假回来的时候把它们拿起来。之后，我们会保持一个稳定的流到NPM @ 5释放。

我们仍在努力寻找更新和更好的方法来明确我们在做什么，并得到您的反馈。我们期待您的输入这些新提案，我相信它会帮助我们产生更好的新理论。

感谢您一直以来的关注和关注。祝您全家节日快乐，新年愉快！
